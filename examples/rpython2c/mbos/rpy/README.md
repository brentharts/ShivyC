# mbos rpython render path

This directory is roadmap **step 5**: the DOM parser and renderer are written in
rpython and translated to C by `tools/py2c.py`, then linked into the mbos kernel
in place of the hand-written `html.c` / `render.c` / `dom.c`. It is the point
where mbos and the Wayland minibrowser converge on the *same* `dom.py` Node
model -- one runs under a compositor, the other on bare metal.

```
cd ..            # examples/rpython2c/mbos
make rpython          # transpile + build build/mbos_rpy.elf
make rpython-test     # headless self-test (embedded page) + screenshot
make rpython-test-net # fetch a page over ARP+UDP, render it, assert, pcap
```

The gcc-C path (`make` / `make test`) is untouched and still builds the same
binary as before; this is an alternate render path, selected by `-DMBOS_RPYTHON`.

## The pieces

| file | role | mirrors |
|---|---|---|
| `dom.py` (copied from `../../minibrowser/dom.py`) | the Node model | itself -- shared verbatim with the Wayland browser |
| `htmlparse.py` | HTML -> Node tree | `html.c`, and `www2json.py` on the host |
| `render.py`    | Node tree -> VGA text via the glue FFI | `render.c`, and `json2qt.py`'s traversal |
| `mbmain.py`    | entry: fetch page from C, parse, render | `json2qt.py` main() |
| `mbos_glue.c`  | the `mbos_glue` FFI shim -> console.c | `mb_ffi.c` / rpyqt's Wayland glue |
| `rt_freestanding.c` + `freestanding_inc/` | freestanding libc for the generated runtime | (new) |
| `gen/` | py2c output (gitignored; regenerated by `make rpython`) | |

`render.py` reaches the console exactly the way `json2qt.py` reaches its canvas:
a `ctypes.CDLL("mbos_glue")` guarded by `sys.implementation.name == 'shivyc'`,
so py2c lowers `_g.mb_putc(...)` to a plain `extern`/`mb_putc(...)` call, and the
same source still imports under CPython (used for the cross-checks below).

## Making py2c output run freestanding

The generated `shivyc_rt.c` is written against a hosted libc (malloc, strlen,
sprintf, setjmp, ...) and a 1 GiB static arena. Rather than edit generated code,
mbos shadows the four hosted headers it pulls (`stdio/stdlib/string/setjmp` +
`ctype`) from `freestanding_inc/`, implements just the reached symbols in
`rt_freestanding.c` (arena-backed malloc, freestanding str/mem, a tiny
`%ld/%s/%p` printf to serial, `strtol`; floats/exceptions are stubbed to fail
loudly), and shrinks the runtime arena with `-DSHIVYC_ARENA_LOG2=20` (1 MiB) so
it fits a kernel loaded at 1 MiB.

## The 32-bit boxing bug (worth knowing)

py2c emits cross-module calls with K&R prototypes, e.g. `extern void Node_add();`,
and relies on the *call site* already holding an `obj` for the by-value argument
(minijson.py does: its `parse_node` returns `obj`). An early version of
`htmlparse.py` kept the freshly built element in a concrete `Node` local and
passed it straight to `top.add(el)`. With no prototype, the compiler passed a
bare 4-byte `Node*` where a 12-byte `obj` was expected -- harmless at 64-bit (the
value rode in registers), silently corrupting the child list at 32-bit. The fix
mirrors minijson: a `_mk(tag) -> "obj"` helper returns the node as an `obj`, so
the value crossing into `Node_add` is the full boxed struct. If you add code
here that stores Nodes into lists or passes them across module boundaries, keep
them `obj`-typed at that boundary.

A fast way to reproduce/debug this class of issue without QEMU: compile the
`gen/*.c` + `rt_freestanding.c` into a static 32-bit ELF with a tiny syscall
`_start` and run it under gdb -- the same corruption shows up as a null-pointer
deref in `Node_tag`, one step from the root cause.
